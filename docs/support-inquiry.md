## 質問1: APIゲートウェイからバックエンドへの通信における送信元IPアドレスについて

APIゲートウェイがバックエンド（Serviceの「ホスト」に指定したエンドポイント）にリクエストを転送する際、送信元となるIPアドレスについて以下を確認させてください。

1. 送信元IPアドレスは固定でしょうか。それとも動的に変わる可能性がありますでしょうか。
2. 送信元IPアドレスのレンジ（CIDR等）は公開されていますでしょうか。
3. 今後、送信元IPアドレスが変更される可能性はございますでしょうか。

バックエンド側にファイアウォールを設定し、APIゲートウェイからのリクエストのみを許可する構成を検討しております。送信元IPが特定できればシンプルなネットワークレベルの制限で実現できるため、この情報が設計上大変重要となります。

## 質問2: APIゲートウェイからプライベートネットワーク内のサーバへの接続について

APIゲートウェイのバックエンドとして、さくらのクラウドのプライベートネットワーク（スイッチ接続）内にあるサーバを指定することは可能でしょうか。以下のケースについてご確認いただけますと幸いです。

### ケースA: VPCルータのNAT経由での接続

VPCルータでスタティックNATを設定し、グローバルIPをプライベートネットワーク内のサーバに紐づけている場合、APIゲートウェイのServiceホストにそのグローバルIPを指定すれば、プライベートネットワーク内のサーバにリクエストは到達しますでしょうか。

```
[APIゲートウェイ] →（グローバルIP）→ [VPCルータ/NAT] →（プライベートIP）→ [サーバ]
```

また、この場合の通信経路はさくらのクラウド内部ネットワークを経由するのでしょうか。それとも一度インターネットを経由する形になりますでしょうか。

### ケースB: プライベートネットワークへの直接接続

APIゲートウェイをスイッチやVPCルータに直接接続し、プライベートネットワーク内のサーバにグローバルIPを経由せずアクセスする機能はございますでしょうか。AWSにおけるAPI GatewayのVPC Link機能に相当するものを想定しております。

### ケースC: 今後の対応予定

上記のような閉域接続の機能が現時点で未対応の場合、将来的に対応される予定はございますでしょうか。

現在の設計ではバックエンドサーバにグローバルIPを付与して公開する構成を検討しておりますが、セキュリティの観点から、できればプライベートネットワーク内に閉じた構成にしたいと考えております。

## 質問3: リクエスト変換における処理順序について

ルートのリクエスト変換設定において、`remove`（ヘッダの削除）と`add`（ヘッダの追加）を同一ヘッダ名に対して同時に設定した場合の挙動を確認させてください。

たとえば、以下のようにリクエスト変換を設定した場合を想定しております。

```json
{
  "remove": {
    "headerKeys": ["X-Custom-Header"]
  },
  "add": {
    "headers": [{"key": "X-Custom-Header", "value": "secret-value"}]
  }
}
```

確認したい点は以下の3つです。

1. クライアントが `X-Custom-Header: fake-value` というヘッダを付けてリクエストを送信した場合、バックエンドに届くヘッダは `X-Custom-Header: secret-value` となりますでしょうか。
2. 処理の順序は `remove`（削除）→ `add`（追加）の順で実行されますでしょうか。
3. `add` はドキュメント上「同名のKeyがリクエストに存在しない場合に追加」とありますが、先に `remove` で削除された後であれば `add` による追加が実行されるという理解で正しいでしょうか。

APIゲートウェイを経由したリクエストであることをバックエンド側で検証する仕組みを検討しております。具体的には、APIゲートウェイでバックエンドへの転送時にシークレットヘッダを付与し、クライアントから送信された同名の偽装ヘッダは事前に削除するという方式です。この方式の実現可否を判断するために、上記の挙動を確認させていただければ幸いです。
